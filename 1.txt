 -----1. Overlap Communication with Computation (Biggest Win)
Your current DLinear::forward does sync_async() + wait_on_stream(0) for row-parallel layers—this is good, but the AllReduce still blocks the compute stream before the next layer starts. The Domino-style overlap technique can pipeline this:

Attention c_proj AllReduce can overlap with LayerNorm ln_2 + MLP fc1 compute
MLP fc2 AllReduce can overlap with ln_1 of the next layer + next layer's c_attn
This requires restructuring the forward pass so the AllReduce of one sub-block runs on the NCCL stream while the compute of the next sub-block runs on stream 0, with dependency events inserted only where truly needed (i.e., at the residual addition). This is the approach described in the Domino architecture you've studied previously.

 5. Use FlashAttention / Fused SDPA
Your attention (lines 104-122) does several separate kernels:

matmul(q, k^T) → 2. mul(scale) → 3. tril(mask) → 4. softmax → 5. matmul(probs, v)
This is 5 separate kernel launches with intermediate materializations (especially the full [B, n_heads, T, T] attention matrix). A fused SDPA kernel (like FlashAttention) would:

Reduce this to 1 kernel
Avoid materializing the O(T²) attention matrix
Reduce memory bandwidth by ~10×
This is probably the single biggest compute optimization if you can integrate cuDNN's fused attention or a FlashAttention-style custom kernel.

 7. Gradient Accumulation Without Graph Rebuilding
Each micro-step currently rebuilds the entire autograd graph. If your framework supports a no_sync() mode (like PyTorch's DistributedDataParallel.no_sync()), you can skip the backward hooks' AllReduce for all micro-steps except the last, avoiding redundant communication during accumulation.
